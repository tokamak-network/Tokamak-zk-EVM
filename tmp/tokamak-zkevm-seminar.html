Motivation

왜 EVM 실행을 증명하려 하는가

Scalability

모든 검증자가 모든 실행을 다시 수행하지 않고, 짧은 proof로 실행 결과를 확인한다.

  • rollup-style validation
  • repeated verifier work 감소
  • 많은 실행을 하나의 검증 흐름으로 압축

Privacy

실행의 정당성은 보이되, 실행 입력의 일부는 숨긴다.

  • private input과 public input 분리
  • private state, shielded account, encrypted execution
  • correctness without full disclosure

응용 목표가 scalability이든 privacy이든, 공통의 기술 문제는 EVM 실행을 검증 회로로 바꾸는 것이다.

Verification View

검증 회로는 실행 경로를 찾지 않고 확인한다

후보 실행 경로branch, loop, call 결과가 이미 정해짐
검증 회로그 경로가 규칙을 만족하는지 확인
accept / reject계산을 다시 찾지 않고 검사

왜 회로 모양이 달라질 수 있는가

  • 실행 branch가 바뀌면 검사해야 할 opcode가 달라진다.
  • 반복 횟수가 바뀌면 필요한 block copy 수가 달라진다.
  • call 구조가 바뀌면 연결해야 할 값의 경계가 달라진다.

같은 bytecode라도 성공 실행 경로가 달라지면 함수 특화 검증 회로의 모양도 달라질 수 있다.

공통점은 이미 관측된 실행을 검증 가능한 입력으로 만든다는 점이고, 차이는 그 입력을 어떤 검증 회로 구조로 처리하는가에 있다.

Two Compilers

하나는 블록을 준비하고, 하나는 실행 기록을 회로 입력으로 바꾼다

qap-compiler

  • 구체적인 EVM 실행과 무관하게 먼저 실행된다.
  • 재사용 가능한 subcircuit library를 만든다.
  • library version이 같으면 여러 함수 실행에서 공유된다.

synthesizer

  • 하나의 EVM 트랜잭션의 실행 기록을 따라간다.
  • 필요한 placement와 witness 값을 만든다.
  • 같은 값이어야 하는 wire들을 value connection으로 기록한다.

Mini Example

하나의 값은 여러 블록을 지나가며 연결된다

실행 기록에서 보이는 일

  • calldata에서 amount를 읽는다.
  • amount가 0이 아닌지 검사한다.
  • amount가 balance update에 다시 사용된다.

검증 회로에서 생기는 일

  • calldata block이 amount 값을 낸다.
  • nonzero check block이 같은 amount를 받는다.
  • accounting block도 같은 amount를 받도록 연결된다.

값 연결은 “같은 논리적 값”이 여러 block copy에서 일관되게 사용되었음을 검증한다.

Reuse Condition

같은 회로를 재사용하려면 성공 실행의 모양이 같아야 한다

같아야 하는 것

  • 같은 종류의 블록
  • 같은 개수의 블록 복사본
  • 같은 값 연결 패턴

왜 필요한가

trusted circuit은 이미 검토되고 verifier workflow에 받아들여진 고정 회로다. 매번 새로 생성된 회로 구조를 on-chain에서 검토한다고 보면 안 된다.

값은 달라져도 된다. 성공 실행에서 필요한 회로의 모양은 같아야 한다.

21 / 29
Contract Shapes

입력이 달라도 성공 실행의 모양이 같으면 재사용 가능하다

안전한 형태

고정 개수의 note를 mint하고 항상 같은 검사 블록을 사용한다.

위험한 형태

입력 배열 길이에 따라 성공 실행의 loop count와 block 수가 달라진다.

조건부 가능

branch가 있어도 모든 성공 경로가 같은 블록 수와 값 연결로 수렴하면 가능하다.

같은 trusted circuit reuse 논의는 성공한 실행 경로만 대상으로 한다. 실패한 트랜잭션 경로는 별도 범위다.

Failure Modes

잘못된 회로 모양을 허용하면 검증 조건이 바뀐다

wrong shape

검토된 회로와 다른 블록 구성이 proof 대상이 될 수 있다.

nondeterminism

같은 실행 기록에서 서로 다른 블록 배치가 나오면 verifier가 무엇을 신뢰하는지 흐려진다.

unsafe fallback

지원하지 않는 모양을 padding이나 skip으로 숨기면 검증 조건이 바뀐다.

지원하지 않는 성공 실행 모양은 조용히 보정하지 말고 명확히 거부되어야 한다.

Backup

CALL 구조는 새로 추측하지 않고 실행 기록을 따라간다

parent contextcalldata를 memory에 준비
child contextcallee의 local EVM steps
parent contextreturn data를 다시 연결

핵심 답변

  • call tree를 새로 추측하지 않는다.
  • 실행 기록에 나타난 call depth와 return boundary를 따라간다.
  • calldata copy와 return-data copy는 값 연결로 검증된다.
  • call count, target, return shape가 변하면 회로 모양도 변할 수 있다.
