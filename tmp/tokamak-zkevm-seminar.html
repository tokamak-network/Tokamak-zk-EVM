Motivation

왜 EVM 실행을 증명하려 하는가

Scalability

모든 검증자가 모든 실행을 다시 수행하지 않고, 짧은 proof로 실행 결과를 확인한다.

  • rollup-style validation
  • repeated verifier work 감소
  • 많은 실행을 하나의 검증 흐름으로 압축

Privacy

실행의 정당성은 보이되, witness의 일부는 숨긴다.

  • private input과 public statement 분리
  • private state, shielded account, encrypted execution
  • correctness without full disclosure

응용 목표가 scalability이든 privacy이든, 공통의 기술 문제는 EVM 실행을 검증 가능한 arithmetic statement로 바꾸는 것이다.

NP View

회로는 답을 찾는 장치가 아니다

NP-style verification

  • prover는 witness가 존재한다고 주장한다.
  • circuit은 그 witness가 statement를 만족하는지 확인한다.
  • 회로의 목적은 search가 아니라 verification이다.

EVM replay에서 달라지는 것

  • branch choice
  • loop count
  • memory/storage access
  • call structure
  • intermediate dataflow

같은 bytecode라도 찾은 실행 trace가 달라지면 검증 회로의 모양도 달라질 수 있다.

Two Compilers

두 compiler는 서로 다른 질문에 답한다

Library compiler: qap-compiler

  • 특정 replay 전에 실행된다.
  • 어떤 reusable constraint components가 있는지 정한다.
  • library version이 바뀌기 전까지 여러 실행에 재사용된다.
  • 최종 transaction circuit을 통째로 만드는 것이 아니다.

Replay compiler: synthesizer

  • 하나의 concrete replay에 대해 실행된다.
  • 이 replay의 값과 value connection을 기록한다.
  • output은 replay-specific이다.
  • reusable subcircuit library를 다시 정의하지 않는다.

Replay To Circuit

EVM replay가 block layout과 value connections를 만든다

block layout

  • replay가 다음 operation을 말해준다.
  • synthesizer는 맞는 reusable block을 고른다.
  • block은 local relation을 검사한다.

value connections

  • 값이 어디서 와서 어디로 가는지 추적한다.
  • stack, memory, calldata, storage, return data를 따라간다.
  • 같은 값이어야 하는 wire들을 연결한다.

Reuse Condition

왜 같은 trusted circuit을 재사용할 수 있어야 하는가

on-chain verifier의 관점

verifier는 고정된 public statement와 proving key에 대한 proof를 확인한다.

임의로 새로 생성된 synthesizer output 자체를 매번 on-chain에서 검증한다고 보면 안 된다.

reuse rule

하나의 trusted circuit은 같은 contract entry의 모든 성공 호출에서 다음이 같을 때 재사용된다.

  • 같은 종류의 block
  • 같은 개수의 block
  • 같은 value connection pattern

기술적으로는 동일한 placement topology와 호환되는 public/private interface layout이 필요하다.

20 / 28
Contract Shapes

어떤 contract entry가 같은 circuit을 재사용하기 쉬운가

Likely reusable

  • fixed-arity minting
  • fixed-shape token transfer
  • bounded single-slot update
  • validation 후 하나의 successful path만 허용

Likely not reusable

  • recipient 수가 loop count를 바꾸는 batch transfer
  • variable-length array processing
  • 성공 branch마다 external call 수가 다른 함수
  • router-like dynamic call target

입력 의존 branch가 있어도 모든 successful path의 circuit shape가 같으면 같은 trusted circuit을 사용할 수 있다.

Failure Modes

잘못된 circuit shape를 신뢰하면 무엇이 깨지는가

wrong shape

verifier가 생각한 successful path와 다른 회로 모양을 증명할 수 있다.

nondeterminism

같은 replay에 대해 서로 다른 block layout이나 value connections가 나올 수 있다.

unsafe fallback

unsupported shape를 조용히 padding, skip, reroute하면 statement가 바뀐다.

Unsupported shape는 명확히 거부되어야 하며, fallback이 statement 변경을 숨기면 안 된다.

Summary

오늘의 핵심 정리

Derivation

  • replay는 이미 찾은 실행 기록이다.
  • synthesizer는 reusable blocks를 고른다.
  • value connections가 copy constraints를 만든다.

Reuse

  • 같은 bytecode가 항상 같은 circuit을 뜻하지 않는다.
  • successful circuit shape가 같을 때만 같은 trusted circuit을 재사용한다.
  • Tokamak zk-EVM은 모든 smart contract에 자동 적용되는 모델이 아니다.

Values may vary; the successful circuit shape must not.

Q&A Backup

복잡한 EVM call 구조를 Synthesizer가 다루는 방식

depth 0Contract A: parent memory에서 calldata 준비
depth 1Contract B: child context의 local EVM steps
depth 0return data가 parent memory로 복사됨

핵심 답변

  • call tree를 새로 추측하지 않는다.
  • interpreter가 만든 concrete message-call replay를 따라간다.
  • context boundary에서 값 복사와 caller semantics를 확인한다.
  • call count, depth, target, return-data shape가 변하면 circuit shape도 변할 수 있다.
